# Как работать с этим шаблонным репозиторием

Этот репозиторий нужен для учебных задач по Python. Он уже подготовлен так, чтобы вы могли сосредоточиться на решении, а не на настройке проекта с нуля. В шаблоне уже есть базовая структура, тесты, проверки качества кода, CI, шаблон Pull Request и базовые ограничения на работу с основной веткой.

## Что здесь уже настроено

В шаблоне уже есть:

- базовая структура проекта;
- проверки качества кода;
- тесты;
- CI;
- шаблон Pull Request;
- базовые ограничения на работу с основной веткой.

Это значит, что вам не нужно вручную собирать окружение проекта по частям, выбирать линтеры или придумывать, как запускать проверки. Всё важное уже задано заранее.

## Что нужно знать в самом начале

### Python в проекте

Шаблон рассчитан на **Python 3.13**.

При этом специально вручную ставить Python 3.13 для этого проекта не нужно. В проекте используется `uv`, и именно он отвечает за окружение и зависимости. Даже если у вас на компьютере уже установлен Python другой версии, это не страшно: `uv` сам подберёт и при необходимости установит подходящую версию Python для виртуального окружения проекта.

### Главный инструмент проекта

В этом шаблоне используется `uv`.

Он нужен для того, чтобы:

- создать и использовать виртуальное окружение;
- установить зависимости проекта;
- запускать команды внутри проекта.

Проверьте, установлен ли `uv`:

```bash
uv --version
```

Если `uv` ещё не установлен, поставьте его так:

```bash
python -m pip install uv
```

## С чего начать после получения репозитория

Обычный старт выглядит так: вы получаете репозиторий, клонируете его, устанавливаете зависимости, включаете `pre-commit`, убеждаетесь, что шаблон живой, а потом уже начинаете писать своё решение.

### 1. Склонируйте репозиторий

```bash
git clone <url-репозитория>
cd <имя-репозитория>
```

### 2. Установите зависимости

```bash
uv sync --group dev
```

Эта команда делает сразу несколько полезных вещей:

- создаёт окружение `.venv`, если его ещё нет;
- подбирает подходящий Python для проекта;
- устанавливает зависимости из `pyproject.toml`, включая инструменты разработки.

### 3. Подключите `pre-commit`

```bash
uv run pre-commit install
```

После этого проверки будут автоматически запускаться перед каждым `git commit`.

### 4. Убедитесь, что всё в порядке

Для локальной проверки перед работой и перед отправкой решения используйте одну основную команду:

```bash
uv run pre-commit run --all-files
```

А для тестов отдельно:

```bash
uv run pytest
```

Если обе команды проходят, значит базовая инфраструктура проекта работает нормально.

## Как устроен шаблон

В шаблоне используется **flat layout**, то есть без директории `src/`.

Пример структуры:

```text
your-repository/
├── pyproject.toml
├── .pre-commit-config.yaml
├── README.md
├── INSTRUCTIONS.md
├── YOUR_APP_NAME_RENAME_ME/
│   ├── __init__.py
│   ├── __main__.py
│   └── main.py
└── tests/
    ├── __init__.py
    └── test_example.py
```

Это значит, что основной код лежит прямо в верхнеуровневом пакете, а тесты — отдельно в `tests/`.

## Где писать своё решение

После клонирования репозитория вам нужно первым делом привести шаблон к виду своего проекта.

### Пакет с кодом

В репозитории есть директория:

```text
YOUR_APP_NAME_RENAME_ME
```

Её нужно переименовать в имя своего проекта.

Важно:

- имя пакета должно быть в `snake_case`;
- основной код нужно хранить внутри этого пакета;
- не нужно складывать `.py`-файлы россыпью в корень репозитория.

Пример:

```text
my_amazing_solution/
├── __init__.py
├── __main__.py
├── main.py
└── service.py
```

### Тесты

Тесты пишутся в директории `tests/`.

Примеры:

```text
tests/test_main.py
tests/test_service.py
```

### Что считается хорошей структурой

Ожидается, что решение будет аккуратно разбито на логические части. Не стоит держать всё приложение в одном огромном файле. Лучше разносить код по модулям и давать понятные имена файлам, функциям, классам и переменным.

## Как запускать проект

В шаблоне уже есть файл `__main__.py`. Это сделано специально, чтобы проект было удобно запускать как модуль.

Допустим, вы переименовали директорию `YOUR_APP_NAME_RENAME_ME` в `my_amazing_solution`. Тогда запуск будет таким:

```bash
uv run python -m my_amazing_solution
```

Именно этот способ запуска для шаблона считается основным.

Почему так удобно:

- команда остаётся короткой;
- импортный контекст корректный;
- структура проекта не расползается;
- запуск одинаково понятен и вам, и преподавателю, и CI.

### Что сейчас лежит в `__main__.py`

В шаблоне уже есть минимальный пример:

```python
if __name__ == "__main__":
    print("Hello, World!")
```

То есть после переименования пакета вы можете сразу проверить, что модульный запуск работает, а потом уже заменить это на свою реальную точку входа.

## Какие правила кода здесь действуют

Ниже не рекомендации «по желанию», а реальные требования шаблона. Они поддерживаются настройками проекта и автоматическими проверками.

### Совместимость с Python

Код должен быть совместим с **Python 3.13**.

### Аннотации обязательны

Аннотации типов обязательны:

- для всех функций;
- для всех методов;
- для тестов;
- для аргументов;
- для возвращаемых значений.

Если функция типизирована только частично, это тоже считается проблемой.

`self` и `cls` отдельно аннотировать не нужно.

### Форматирование и стиль

В шаблоне используются:

- `ruff check`;
- `ruff format`;
- `mypy`;
- `pytest`;
- `pre-commit`.

Что важно понимать заранее:

- здесь используется `ruff format`;
- строки приводятся к **двойным кавычкам**;
- импорты приводятся к единому порядку;
- длина строки ориентирована на **100 символов**;
- докстринги не обязательны;
- `print` в учебных задачах использовать можно.

### Что реально проверяется автоматически

Автоматические проверки отслеживают в том числе:

- ошибки стиля;
- ошибки импортов;
- неиспользуемые переменные и импорты;
- проблемы с именованием;
- часть потенциально опасных конструкций;
- отсутствие аннотаций;
- несовременный typing-стиль;
- ошибки типизации;
- падение тестов.

## Что делает `pre-commit`

Когда вы делаете коммит, `pre-commit` запускает проверки автоматически. В этом шаблоне он отвечает не только за Python-код, но и за общую аккуратность репозитория.

Перед коммитом автоматически запускаются:

- проверка служебных файлов;
- исправление лишних пробелов;
- исправление отсутствия перевода строки в конце файла;
- `ruff check --fix`;
- `ruff format`;
- `mypy`.

Если какая-то проверка не прошла, это не повод коммитить «как есть». Нужно сначала исправить проблему и только потом повторить коммит.

Если хотите прогнать всё руками заранее по всему проекту, используйте:

```bash
uv run pre-commit run --all-files
```

## Как обычно выглядит работа над задачей

Нормальный рабочий процесс здесь такой:

1. Получить репозиторий.
2. Склонировать его к себе.
3. Установить зависимости через `uv`.
4. Переименовать пакет `YOUR_APP_NAME_RENAME_ME` в имя своего проекта.
5. Создать отдельную ветку от `main`.
6. Написать код и тесты.
7. Проверить проект локально.
8. Сделать коммит.
9. Запушить ветку на GitHub.
10. Открыть Pull Request в `main`.
11. При необходимости запросить ревью через Telegram-бота.

Это важный момент: работа идёт не в `main`, а в отдельной ветке.

Пример создания ветки:

```bash
git checkout -b feature/add-solution
```

После изменений обычный сценарий выглядит так:

```bash
git add .
git commit -m "Добавил решение задачи"
git push origin feature/add-solution
```

## Что нельзя делать

### Нельзя работать напрямую в ветку `main`

Ориентируйтесь на простое правило: **не разрабатывайте в `main`**.

То есть:

- не коммитьте в ветку `main`;
- не ведите разработку в ветке `main`;
- все изменения делайте в отдельной ветке;
- сдавайте задачу через Pull Request.

Даже если GitHub где-то технически позволит сделать лишнее действие, правильный процесс всё равно такой:

**ветка -> PR -> проверки -> review**

### Нельзя игнорировать проверки

Не отправляйте код на ревью преподавателю который:

- не проходит линтеры и форматирование;
- не проходит типизацию;
- не проходит тесты.

Практически это означает, что перед отправкой нужно прогнать:

```bash
uv run pre-commit run --all-files
uv run pytest
```

### Нельзя коммитить мусор

Не добавляйте в репозиторий:

- `.venv`;
- `__pycache__`;
- кэши тестов;
- временные файлы редактора;
- случайные отчёты и артефакты.

## Как сдавать задачу через GitHub

После того как ветка отправлена на GitHub, задача обычно считается готовой к сдаче не в момент `push`, а в момент открытия Pull Request.

Что нужно сделать:

1. Открыть репозиторий на GitHub.
2. Создать Pull Request в `main`.
3. Заполнить шаблон PR.
4. Убедиться, что описание PR нормальное и содержательное.

Хорошее описание PR обычно отвечает на три вопроса:

- что сделано;
- зачем это сделано;
- как это проверялось.

Ориентир по объёму описания PR — примерно **100 символов и больше**, но важнее не длина, а содержательность.

## Какие ограничения и проверки обычно действуют при merge

По умолчанию в учебном процессе и в стандартной policy этого шаблона ориентируйтесь на такую модель:

- merge идёт через Pull Request;
- в основной ветке ожидается зелёный CI;
- базовые обязательные проверки обычно `lint` и `test`;
- обычно нужен минимум **один approve**;
- после нового `push` в PR старые approve могут устареть;
- незакрытые обсуждения в PR могут блокировать merge;
- review может требоваться именно от преподавателя или назначенного ревьюера.

Проще говоря: сначала приводите проект в порядок локально, потом открывайте PR, и только после этого просите ревью.

## Как это связано с Telegram-ботом

В учебной инфраструктуре используется Telegram-бот. Обычно путь студента выглядит так:

1. Пройти регистрацию через `/start`.
2. Быть назначенным в группу.
3. Получить доступ к своему учебному репозиторию.
4. Сделать изменения в отдельной ветке.
5. Открыть PR.
6. Запросить ревью через бота.

Важно помнить:

- для ревью и LLM-review нужен **открытый PR**;
- если вы ещё не назначены в группу, основные student-команды могут быть недоступны.

## Чек-лист перед отправкой

Перед тем как отправлять задачу, быстро проверьте себя:

- код лежит внутри вашего верхнеуровневого пакета;
- пакет переименован из `YOUR_APP_NAME_RENAME_ME`;
- тесты лежат в `tests/`;
- функции и тесты типизированы;
- используются современные аннотации;
- код не лежит россыпью в корне;
- `uv run pre-commit run --all-files` проходит;
- `uv run pytest` проходит;
- проект запускается через `uv run python -m <имя_вашего_пакета>`;
- Pull Request создан в `main`;
- описание PR заполнено по делу.

## Короткий сценарий целиком

Если хочется увидеть весь путь в одном месте, то он выглядит так:

```bash
git clone <url-репозитория>
cd <имя-репозитория>

uv sync --group dev
uv run pre-commit install

git checkout -b feature/my-task

# переименуйте YOUR_APP_NAME_RENAME_ME в имя своего пакета
# пишите код внутри своего пакета верхнего уровня
# пишите тесты в tests/

uv run python -m my_amazing_solution
uv run pre-commit run --all-files
uv run pytest

git add .
git commit -m "Реализация задачи"
git push origin feature/my-task
```

После этого:

1. создайте Pull Request в `main`;
2. заполните шаблон PR;
3. запросите ревью через бота, если это требуется вашим процессом.

## Если требования расходятся

Иногда у конкретной задачи могут быть дополнительные условия от преподавателя. В таком случае ориентируйтесь по приоритету.

Если есть расхождение между:

- этой инструкцией;
- текстом конкретного задания;
- замечанием преподавателя;
- требованиями в Pull Request;

то порядок такой:

1. явные требования преподавателя к конкретной задаче;
2. технические ограничения репозитория и CI;
3. эта инструкция.

## Если с Git стало тяжело

Если вы чувствуете, что запутались в ветках, коммитах, ребейзах, PR или вообще в базовом Git-flow, не пытайтесь просто «протыкать кнопки наугад». Гораздо выгоднее один раз нормально разобраться.

Вот очень полезная интерактивная игра по Git:

https://learngitbranching.js.org/

На неё правда стоит потратить 20–40 минут жизни и пройти её целиком. После этого работа с ветками и PR обычно становится намного понятнее.
